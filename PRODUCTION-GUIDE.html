一、当前状态评估

1.1 做得好的地方 ✅

  • 分层清晰 — Domain / Protocol / Collection / Forwarder / Transport / Storage 各司其职
  • Result 模式 — 全项目 OperationResult<T> 而非异常驱动控制流
  • 接口隔离 — IProtocolDriver / IMeasurementStore / IForwardBuffer 都有清晰边界
  • 两阶段提交 — ForwardBuffer Enqueue→Dequeue→Commit，断电不丢
  • BackgroundService — 采集引擎生命周期由框架管理，而非手写线程
  • 死区过滤 — 有 Deadband 概念，减少无意义上报
  • 健康监控 — DeviceHealthMonitor 连续失败检测 + 自动 Offline/恢复
  • 前端面板 — Vue 3 + SignalR，有基础的设备管理和实时监控

1.2 当前覆盖度雷达

维度覆盖度关键缺口

二、P0 — 安全（阻塞上线）P0

生产环境公网或企业内网部署，安全是绝对底线。没有认证、没有加密，绝对不允许上线。
2.1

API 认证 — JWT Bearer Token

  • ☐ 添加 Microsoft.AspNetCore.Authentication.JwtBearer NuGet 包
  • ☐ 实现登录接口 POST /api/auth/login，返回 access_token + refresh_token
  • ☐ 配置 AddAuthentication().AddJwtBearer()
  • ☐ Controller 添加 [Authorize] 特性
  • ☐ SignalR Hub 连接时校验 Token：OnConnectedAsync 中读取 access_token query param
  • ☐ Token 过期刷新机制
2.2

MQTT TLS

  • ☐ MqttConnectionOptions 增加 TLS 配置：CA 证书路径、客户端证书、是否忽略证书错误
  • ☐ MqttClientWrapper 在 Connect 时使用 MqttClientOptionsBuilderTlsParameters
  • ☐ 生产默认要求 TLS 1.2+，开发环境允许不加密
2.3

配置与凭证外部化

  • ☐ 所有连接字符串、密钥从 appsettings.json + 环境变量读取，零硬编码
  • ☐ Program.cs 中 builder.Configuration.AddEnvironmentVariables("NITRO_")
  • ☐ 敏感字段（MQTT 密码、JWT 密钥）支持通过文件挂载读取（Docker Secrets / K8s）
2.4

审计日志

  • ☐ 实现 AuditMiddleware：记录每次 API 调用的 User、IP、Action、Timestamp、Result
  • ☐ 敏感操作（设备增删、点位修改、状态变更）记录到 audit_log 表

三、P1 — 容错与弹性 P1

3.1

重试策略 — Polly

// 在 DeviceReader 中包装驱动调用
services.AddResiliencePipeline("device-read", builder =>
{
    builder.AddRetry(new RetryStrategyOptions
    {
        MaxRetryAttempts = 3,
        BackoffType       = DelayBackoffType.Exponential,
        Delay             = TimeSpan.FromMilliseconds(200),
        MaxDelay          = TimeSpan.FromSeconds(5),
        ShouldHandle      = args => args.Outcome.Exception is not OperationCanceledException
                                     ? PredicateResult.True()
                                     : PredicateResult.False()
    });
});
  • ☐ DeviceReader.ReadDeviceAsync 通过 Polly pipeline 调用驱动，失败自动重试
  • ☐ Forwarder MQTT Publish 失败重试（区分网络错误和业务错误）
  • ☐ 重试计数计入 DeviceHealthMonitor，与熔断联动
3.2

熔断器 — Circuit Breaker（每设备独立）

// 每设备独立熔断器注册表
public class CircuitBreakerRegistry
{
    private readonly ConcurrentDictionary<Guid, CircuitBreakerPolicy> _breakers = new();

    public CircuitBreakerPolicy GetOrCreate(Guid deviceId) { /* ... */ }

    // 配置：连续失败 5 次 → 断开 30s → 半开探测 1 次 → 成功则关闭，失败继续断开

  • ☐ 熔断/恢复事件记录日志 + 触发告警
  • ☐ 熔断器状态通过 Metrics 暴露
3.3

并发行限流

// DeviceCollector 添加信号量
private readonly SemaphoreSlim _concurrencyLimit = new(8); // 最多同时采集 8 台设备

public async Task CollectOnceAsync(CancellationToken ct)
{
    var devices = await _deviceManager.GetByStatusAsync(DeviceStatus.Online, ct);
    var tasks = devices.Value!.Select(async device =>
    {
        await _concurrencyLimit.WaitAsync(ct);
        try { await CollectDeviceAsync(device, ct); }
        finally { _concurrencyLimit.Release(); }
    });
    await Task.WhenAll(tasks);
}
  • ☐ 并发数可配置，默认 = CPU 核心数
  • ☐ 避免 200 台 Modbus TCP 设备同时建立连接
3.4

死信队列 (DLQ)

// IForwardBuffer 扩展
public interface IForwardBuffer

  • ☐ 管理面板可查看死信、手动重放或丢弃
3.5

优雅关闭 (Graceful Shutdown)

// Program.cs
var app = builder.Build();

// 注册生命周期钩子
app.Lifetime.ApplicationStopping.Register(() =>
{
    logger.LogInformation("收到停止信号，开始优雅关闭...");
    // 1. 停止接收新采集任务
    // 2. 等待进行中的采集完成（最长 30s）
    // 3. Flush ForwardBuffer（确保未提交数据落盘）
    // 4. Disconnect MQTT（发送 LWT 遗嘱消息）
    // 5. 释放所有资源
});
3.6

MQTT 断连衔接续传

  • ☐ IMqttClient.StateChanged 事件监听：Connected → 标记在线，Disconnected → 标记离线
  • ☐ 离线期间 Forwarder 数据继续入缓冲（现有 ForwardBuffer 已支持）
  • ☐ 恢复连接后，Forwarder 按限速（如每秒 100 条）追补积压数据
  • ☐ 避免恢复瞬间冲垮 MQTT Broker

5.1

告警引擎 — Alarm 模块落地

// 领域模型
public class AlarmRule
{
    public Guid Id { get; init; }
    public Guid DeviceId { get; init; }

    public string Operator { get; set; }   // ">", "<", ">=", "<=", "==", "!="
    public double Threshold { get; set; }

    public AlarmSeverity Severity { get; set; }  // Info, Warning, Critical

public class Alarm
{
    public Guid Id { get; init; }

    public AlarmSeverity Severity { get; init; }
    public DateTime OccurredAt { get; init; }
    public DateTime? AcknowledgedAt { get; set; }
    public DateTime? ResolvedAt { get; set; }
}
  • ☐ 告警规则 CRUD API
  • ☐ AlarmEvaluator：每当 Dispatcher 产出 PointSnapshot，异步评估关联规则
  • ☐ 告警生命周期：触发 → 确认 → 恢复
  • ☐ 告警通知渠道：MQTT 推送 + Web 弹窗 + （未来）短信/钉钉
5.2

配置热加载

  • ☐ 设备/点位变更后，DeviceManager 发出事件，CollectionEngine 监听并刷新设备列表缓存
  • ☐ 不需要重启服务即可生效
  • ☐ 注意：正在采集的设备不受影响，下一轮采集周期自动使用新配置
5.3

点位批量导入

  • ☐ Excel/CSV 导入：自动创建点位（地址自动计算偏移）
  • ☐ Modbus 模板：选择寄存器类型(HR/IR/Coil) + 起始地址 + 数量 + 数据类型 → 批量生成
  • ☐ OPC UA 模板：指定 NodeId 前缀 → 浏览子树 → 批量导入
5.4

告警通知渠道

  • ☐ IAlarmNotifier 接口，支持多种渠道
  • ☐ MQTT 推送（实时）
  • ☐ Webhook 回调
  • ☐ 钉钉/企业微信 Bot（国内工厂标配）

六、P2 — 数据面增强 P2

6.1

本地预聚合

// 每 1 分钟跑一次
public class AggregateWorker : BackgroundService
{
    // 1. 从 MeasurementStore 读取最近 1 分钟的 PointSnapshot
    // 2. 按 (DeviceId, PointId) 分组计算 min/max/avg/sum/count
    // 3. 写入 aggregate 表（或发送到 MQTT topic nitrogateway/{deviceId}/aggregates）
}
  • ☐ 减少上报数据量：边缘算完分钟级聚合，云端只存聚合结果
  • ☐ 配置维度：1min / 5min / 15min / 1h
6.2

时序存储优化

  • ☐ SqliteMeasurementStore 改用真正的批量 INSERT（多 VALUES 拼接或使用 SqliteBlob）
  • ☐ 按天分表：measurements_20260702，自动创建和清理
  • ☐ PurgeAsync 删除旧表而非 DELETE WHERE（DROP TABLE 远快于 DELETE）
  • ☐ SQLite WAL 模式、synchronous=NORMAL、适当增大 cache_size
6.3

定时清理策略

  • ☐ 在 Scheduler 注册 PurgeTask：每天凌晨清理 N 天前的数据
  • ☐ 配置保留策略：measurements 保留 30 天，aggregates 保留 365 天
  • ☐ 清理时检查磁盘空间，低于阈值暂停写入（边缘设备存储有限）

七、P3 — 多协议 & 写操作 P3

7.1

协议补齐

  • ☐ Modbus RTU — ModbusRtuDriver 代码已有，补齐串口库 + DI 注册 + 配置
  • ☐ OPC UA — 当前只有基础读取，补齐：证书管理、订阅(Subscription)模式、安全策略
  • ☐ Siemens S7 — 国内工厂第一优先级缺失协议。用 Sharp7 或 S7netplus，实现 IProtocolDriver
  • ☐ BACnet（楼宇）和 EtherNet/IP（罗克韦尔）按需补充
7.2

写操作全链路

// API: POST /api/devices/{deviceId}/points/{pointId}/write  { "value": 23.5 }
//   → 权限校验（写操作比读操作更需要审计）
//   → DeviceManager.VerifyWriteAccess(deviceId, pointId)
//   → IProtocolDriver.WriteAsync(point, value)
//   → 审计日志

  • ☐ 写入前权限校验
  • ☐ 写入后读回验证（可选，对关键控制）
  • ☐ 前端写操作 UI

八、P3 — 集群化 P3

8.1

Leader Election（防重复采集）

  • ☐ 多实例部署时，同一台设备只应被一个节点采集
  • ☐ 轻量方案：基于数据库的表锁选举（INSERT OR FAIL INTO leader_lock）
  • ☐ 进阶方案：Redis RedLock 或 etcd
  • ☐ Collector 启动时获取锁，只有持锁节点执行采集
8.2

设备分片

  • ☐ 按 deviceId 一致性哈希分配到不同节点
  • ☐ 节点离线时，其设备自动迁移到其他节点（故障转移）
  • ☐ 这个需求 90% 的边缘部署不需要，先不急着做

九、测试体系 必须

9.1

三层测试策略

单元测试

  • ☐ PointValuePipeline：缩放计算、死区过滤、各 DataType 转换
  • ☐ ModbusAddressParser：各种地址格式解析
  • ☐ DeviceHealthMonitor：阈值触发逻辑
  • ☐ DataDispatcher：双写互不阻塞逻辑

集成测试

  • ☐ Mock Modbus TCP 设备（用 EasyModbus 或手写 TCP 服务器模拟 PLC）
  • ☐ DeviceReader → Pipeline → Dispatcher 完整链路
  • ☐ SqliteForwardBuffer Enqueue → Dequeue → Commit 流程
  • ☐ DeviceHealthMonitor 连续失败触发 Offline 状态变更

端到端测试

  • ☐ 启动完整应用 → 注册设备 → 等待采集 → 验证 SQLite 有数据 → 验证 ForwardBuffer 有数据
  • ☐ Mock MQTT Broker（用 Moq 或内存实现）
快速起步建议：先写一个 Mock Modbus TCP 服务器（几十行代码），然后基于它写一个集成测试覆盖 "注册设备 → 采集 → 存入 SQLite" 这条黄金路径。这个测试的价值远大于 100 个孤立单元测试。
