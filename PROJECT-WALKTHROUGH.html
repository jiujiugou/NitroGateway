场景：工厂车间

一个车间里有一台西门子 PLC（铁柜子）。上面接着三根传感器线：

温度传感器→ 接到 PLC 的寄存器 40001当前值：60.5°C
压力传感器→ 接到 PLC 的寄存器 40003当前值：2.1 bar（PLC 存储为 210，需 ×0.1）
运行开关→ 接到 PLC 的 00001 布尔位当前值：1 开 / 0 关

PLC 通过网线接到工厂局域网，IP 地址是 192.168.1.10:502。

PLC 每几毫秒自己刷新一次寄存器。它不等任何人，也听不懂"把数据给我"——它只说 Modbus 协议。

ModbusTcpDriver 做了两件事：

① 拼帧——发"给我 40001 开始的 5 个寄存器"
// FluentModbus 帮网关拼了这段 Modbus TCP 帧（12 字节） 帧格式: [事务ID][协议ID][长度][站号][功能码 0x03][起始地址 40001][数量 5] ↓ 通过 TCP Socket 发给 PLC PLC 收到，理解"他要读 40001 后面的 5 个寄存器"
② 收帧——拆开 PLC 的回复
// PLC 的回复帧 帧格式: [事务ID][协议ID][长度][站号][功能码 0x03][字节数][16位值][16位值]... [0,1] [0,0] [13] [1] [3] [10] [8字节数据...] // 其中 8 字节数据是: 寄存器地址 原始字节 数据类型 含义 40001 [0x42,0x70, Float [0x4270,0x0000] 0x00,0x00] = 60.0 40003 [0x00,0xD2] Int16 [0x00D2] = 210 40004 [0x00,0x00, Int32 [0x0000,0x0032] 0x00,0x32] = 50 00001 [0x01] Bool [1] = true

FluentModbus 把这些字节转成 ushort[] 返回。然后 ModbusTcpDriver 调用自己的 DecodeRegisters()：

// ModbusProtocolHelper.DecodeRegisters() // 驱动内部把 ushort[] 解码为领域值 ushort[]{0x4270,0x0000} + DataType.Float → 60.0 (double) ushort[]{0x00D2} + DataType.Int16 → 210 (short) ushort[]{0x0000,0x0032} + DataType.Int32 → 50 (int) ushort[]{0x0001} + DataType.Bool → true (bool) // 每个值包进 RawPointValue，返回 4 个 RawPointValue RawPointValue { Point=Temperature, Value=60.0, Timestamp=... } RawPointValue { Point=Pressure, Value=210, Timestamp=... } RawPointValue { Point=FlowRate, Value=50, Timestamp=... } RawPointValue { Point=Running, Value=true, Timestamp=... }
关键文件：ModbusTcpDriver.cs（334 行）、ModbusProtocolHelper.cs（80 行，解码/分组/编码）、DeviceReader.cs（40 行）

// PointValuePipeline.Process() — 对每一个 RawPointValue 做三件事 // ── ① 缩放 ── // Temperature: 60.0 × 1.0 + 0 = 60.0 (Float, ScaleFactor=1.0) // Pressure: 210 × 0.1 + 0 = 21.0 (Int16, ScaleFactor=0.1, PLC 存的是 10 倍值) // FlowRate: 50 × 0.01 = 0.5 (Int32, ScaleFactor=0.01) // Running: true → 直接输出，不缩放 (Bool 类型跳过) // ── ② 死区判定 ── // Temperature: |60.0 − 59.8| = 0.2 < 死区 0.5 → 丢弃！不产生这条 snapshot // Pressure: 没有死区 → 输出 // FlowRate: 没有死区 → 输出 // Running: Bool 不判死区 → 输出 // ── ③ 组装 PointSnapshot ── PointSnapshot { DeviceId:abc-123, DevicePointId:pt-2, RawValue:210, Value:21.0, Quality:Good } PointSnapshot { DeviceId:abc-123, DevicePointId:pt-3, RawValue:50, Value:0.5, Quality:Good } PointSnapshot { DeviceId:abc-123, DevicePointId:pt-4, RawValue:true, Value:true, Quality:Good } // Temperature 那一条因为死区被丢弃了，不输出
为什么需要死区？
温度传感器可能有 ±0.1°C 的噪声。如果没有死区，网关每一秒都会报"温度从 60.0 变成 60.1，又变成 59.9"。死区过滤掉这些微小的抖动，只有真正的变化（>0.5°C）才记录。
关键文件：PointValuePipeline.cs（194 行，纯函数，无 IO）

第六步：健康上报

6HealthReporter 告诉 Monitor："这一轮 3 个成功"
// HealthReporter.Report(deviceId, 3 成功, 0 失败, null) _healthMonitor.ReportSuccess(deviceId); // → SuccessCount++ → 连续 3 次 → 触发 Online 事件 // → IDeviceManager.UpdateStatusAsync(deviceId, Online)
如果某轮采集全部失败（比如网线拔了），ReportFailure 被调，FailCount++，连续 10 次失败 → 触发 Offline → 设备的 Status 从 Online 变成 Offline → 仪表盘上那个绿色标签变成红色。

关键设计：两阶段提交
Dequeue 不移除数据 → 发送成功后才 Commit 删除。发送失败的数据留在 Buffer 里，下一轮重新发。这就是为什么 Buffer 不怕断网。
